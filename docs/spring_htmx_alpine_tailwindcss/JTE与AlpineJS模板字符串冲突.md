# JTE 与 Alpine.js 模板字符串冲突及解决办法

在 **Spring Boot + JTE（Java Template Engine）** 与 **Alpine.js** 同页混用时，两类语法都会用到 **`${ ... }`**：

| 场景 | 含义 |
|------|------|
| **JTE** | `${expression}` 在服务端渲染时求值并输出 |
| **Alpine / 浏览器 JS** | 反引号模板字符串中的 `${expr}` 由浏览器在运行时解析 |

若把 Alpine 的模板字符串直接写进 `.jte` 文件，JTE 会先处理 `${...}`，导致页面里拿到的不是你想交给 Alpine 的那段 JavaScript，从而出现**空白、错字串或运行时报错**。

下面说明原因与常用解决办法。

---

## 冲突原因简述

1. JTE 在服务端解析模板，遇到 `${...}` 会当作 **JTE 表达式**执行。
2. 你希望输出到 HTML 里的，可能是 Alpine 的 **`x-data`、`@click`、内联脚本**里带 **反引号 `` ` ``** 的模板字符串，其中的 **`${...}` 应留给浏览器**。
3. 二者抢同一套 `${}` 写法，就会产生冲突。

---

## 解决办法

### 1. 在 JTE 中转义 `$`（最常用）

JTE 支持用 **反斜杠转义美元符**：在需要**原样输出**给浏览器的地方，把 `$` 写成 `\$`，这样 JTE 不会把它当作自己的插值，最终 HTML 里仍是 Alpine 需要的 `${...}`。

**思路**：凡是「要给 Alpine / 浏览器 JS 用的模板字符串里的 `${`」，在 `.jte` 里写成 `\${`。

示例（概念示意，按你项目里的实际属性名调整）：

```html
<!-- JTE 会输出：@click="console.log(`Hello ${name}`)" -->
<button @click="console.log(`Hello \${name}`)">
  点击
</button>
```

若整段脚本或属性里有多处 `${`，每一处若都要交给浏览器而非 JTE，都需要在 JTE 源文件中转义。

> **注意**：不同 JTE 版本对转义的细节以官方文档为准；若升级 JTE，建议复查「字面量 `$`」的写法是否仍为 `\$`。

---

### 2. 少用浏览器模板字符串，改用 Alpine 表达式

很多场景不必在属性里写 `` `...${}...` ``，可改为：

- **字符串拼接**：`'Hello ' + name` 或 Alpine 的表达式写法。
- **`x-data` 用对象字面量**：`x-data="{ name: 'World' }"`，避免在属性里嵌反引号模板字符串。

这样 JTE 侧不再出现「浏览器才需要的 `${`」，冲突自然消失。

---

### 3. 把 Alpine 逻辑放到独立 `.js` 文件

由 **静态资源**（如 `src/main/resources/static/js/...js`）提供纯 JavaScript，**不经过 JTE 解析**，则文件内可随意使用模板字符串与 `${}`。

页面里只保留 `script src="..."` 与 `x-data` 指向组件数据；JTE 只负责输出 HTML 结构与数据占位（如服务端渲染的变量仍用 JTE 的 `${model.xxx}`，与前端 JS 文件分离）。

---

### 4. 用 `x-html` / 服务端拼好字符串（谨慎）

若必须输出一大段含 `${` 的 HTML/脚本，可由服务端在 JTE 里**拼好整段字符串**再输出，或使用 `x-html`（**务必做好 XSS 防护**，只对可信内容使用）。

一般更推荐前三种方式，避免把可执行逻辑塞进 `x-html`。

---

## 实践建议

1. **先分清**：这一处 `${` 是 **给 JTE 在服务端用的**，还是 **给浏览器里 Alpine/JS 用的**。
2. **给浏览器的那部分**：在 `.jte` 里用 `\$` 转义，或挪到外部 JS，或改写表达式避免模板字符串。
3. **与 HTMX 同用时**：冲突仍发生在「JTE 与内联 JS」之间；HTMX 属性若含 `${`，同样要注意是否被 JTE 抢先解析。

按上述方式即可在 **Spring + JTE + HTMX + Alpine.js + Tailwind CSS** 技术栈中稳定混用模板与前端交互逻辑。
